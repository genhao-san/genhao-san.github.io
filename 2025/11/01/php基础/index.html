1. PHP函数类

1.1 is_numeric()函数

PHP中 is_numeric() 函数的返回值是布尔类型（bool）：若参数是“数字或数字字符串”，返回 true ；否则返回 false 。

关键说明

  • 返回 true 的情况：参数为整数、浮点数、纯数字字符串（如 “123” ）、科学计数法字符串（如 “9e9” ）、带正负号的数字字符串（如 “+123” 、 “-45.6” ）。

  • 示例： is_numeric(123) → true ； is_numeric(“123.45”) → true ； is_numeric(“8e6”) → true 。

  • 返回 false 的情况：参数为非数字相关的字符串（如 “123abc” ）、数组、对象、 null 等。

  • 示例： is_numeric(“abc123”) → false ； is_numeric([1,2]) → false 。

1.2 intval()函数

intval() 是 PHP 中的内置函数，核心作用是将传入的变量（如字符串、浮点数等）转换为整数类型，转换规则遵循“从左到右取有效数字，遇到非数字字符则停止”的逻辑。

  • 字符串转整数：从字符串开头取连续的数字字符，遇到非数字（如字母、符号、空格）则停止；若开头无数字，返回 0。

  • 示例： intval(“123abc”) → 123； intval(“abc123”) → 0； intval(“ 456”) → 456（忽略开头空格）。

  • 浮点数转整数：直接“截断小数部分”（不是四舍五入）。

  • 示例： intval(123.999) → 123； intval(-45.1) → -45。

  • 布尔值转整数： true 转 1， false 转 0。

  • 示例： intval(true) → 1； intval(false) → 0。

  • 数组/NULL 转整数：数组和 NULL 转换后均返回 0。

  • 示例： intval([]) → 0； intval(NULL) → 0。

1.3 preg_match()函数

基本语法

  1. $pattern：必需，要匹配的正则表达式模式（需用定界符包裹，如 /pattern/）。

  2. $subject：必需，要搜索的目标字符串。

  3. $matches

    ：可选，引用传递的数组，用于存储匹配结果。

    • $matches[0] 存储完整匹配的字符串。
    • $matches[1] 及以后存储子组（括号内的分组）的匹配结果。

  4. $flags

    ：可选，匹配标志，常用值：

    • PREG_OFFSET_CAPTURE：使 $matches 中的每个元素不仅包含匹配值，还包含其在字符串中的偏移量（位置）。

  5. $offset：可选，从字符串的指定位置（字节数）开始搜索，默认从 0 开始。

  • 向 preg_match 函数传入数组作为参数时（尤其是 $pattern 或 $subject 参数），会导致函数执行错误并返回 false

2. PHP伪协议

PHP伪协议是PHP支持的特殊文件协议，可用于读取文件、执行代码、操作数据流等，常出现在CTF的文件包含漏洞场景中。

2.1 常用PHP伪协议及核心用途

  • file://：读取本地文件，是最基础的本地文件读取协议，需知道文件绝对/相对路径。

  • 示例： file:///etc/passwd （读取Linux系统密码文件）

  • php://：操作PHP输入输出流，是CTF中最常用的协议组。

  • php://filter ：读取文件源码（常配合 base64-encode 绕过过滤）。

  • 示例： php://filter/read=convert.base64-encode/resource=index.php

  • php://input ：读取POST请求体数据，可配合文件包含执行POST提交的代码（需开启 allow_url_include=On ）。

  • data://：直接执行嵌入的代码，需开启 allow_url_include=On ，类似“代码注入快捷方式”。

  • 示例： data://text/plain;base64,PD9waHAgcGhwaW5mbygpOz8+ （base64解码后为

    ）

  • phar://：读取phar文件内的资源，可利用phar文件的特性绕过文件后缀限制。

  • 示例： phar://test.zip/test.php （读取压缩包内的test.php文件）

2.2 file:// 协议实战案例

  • 使用条件：知道目标文件的绝对/相对路径，无特殊配置要求（默认支持）。
  • 常见绕过技巧：
  • 路径遍历：若过滤了 ../ ，可尝试 ..// （部分服务器会解析为 ../ ）、 ….//….// 等变形。
  • 绝对路径直接读取：当相对路径被限制时，直接用 file:///etc/passwd （Linux）或 file://C:/Windows/system32/drivers/etc/hosts （Windows）。
  • CTF题目场景：某LFI题目仅允许输入 [a-z0-9] ，无法输入 ../ ，直接提交 file:///etc/passwd 读取系统用户信息。

2.3 php://filter 协议实战案例

  • 使用条件：默认支持，无需额外配置，核心用于读取“内容含特殊字符”的文件（如PHP源码）。
  • 常见绕过技巧：
  • Base64编码绕过内容过滤：用 php://filter/read=convert.base64-encode/resource=test.php ，将PHP源码编码为Base64，避开“内容不能含特殊字符”的限制。
  • 多过滤器叠加：若过滤了 base64-encode ，可尝试 convert.quoted-printable-encode 等其他编码过滤器。

2.3.1 编码过滤器

​ 一、核心转换类编码过滤器（CTF高频使用）

​ 这类过滤器可将文件内容转为指定编码格式，适配“内容字符限制”场景，常见如下：

  • convert.base64-encode / convert.base64-decode ：Base64编码/解码，最常用（仅含 A-Za-z0-9+/= ，完美适配字符过滤）。
  • convert.quoted-printable-encode / convert.quoted-printable-decode ：quoted-printable编码/解码，特殊字符转为 =XX （如 <? → =3C=3F ），编码后仅含字母、数字和 = 。
  • convert.iconv.* ：字符集转换过滤器，格式为 convert.iconv.[输入编码].[输出编码] ，可实现多编码转换（如 UTF-8 转 GBK ）。
  • 示例： convert.iconv.UTF-8.UTF-16 （将UTF-8转为UTF-16编码，绕过纯ASCII过滤）。

​ 二、压缩类编码过滤器（辅助绕过）

​ 这类过滤器先压缩内容再编码，可减少内容体积，偶尔用于绕过“内容长度限制”：

  • zlib.deflate / zlib.inflate ：zlib压缩/解压，压缩后内容为二进制，需配合Base64编码使用。
  • bzip2.compress / bzip2.decompress ：bzip2压缩/解压，与zlib类似，需搭配编码过滤器。

​ 三、CTF实战组合示例

​ 当 base64-encode 被过滤时，可组合其他过滤器使用：

  1. 用 quoted-printable-encode 读取PHP源码：
    php://filter/read=convert.quoted-printable-encode/resource=flag.php
    （编码后内容如 =3C=3Fphp=20echo=20’flag{test}’;=3F=3E ，解码后可得源码）
  2. 用 iconv + base64 双重编码：
    php://filter/read=convert.iconv.UTF-8.UTF-7|convert.base64-encode/resource=flag.php
    （先转UTF-7，再Base64，双重绕过关键字过滤）
  • CTF题目场景：某题目用 file_get_contents 读取文件，且验证内容不能含 <? ，提交 php://filter/read=convert.base64-encode/resource=flag.php ，解码后得到含 flag{} 的PHP源码。

2.4 php://input 协议实战案例

  • 使用条件：必须开启 allow_url_include=On ，且后端使用 include / require 函数（而非 file_get_contents ）。
  • 常见绕过技巧：
  • POST数据提交代码：若题目过滤了 （短标签，需开启 short_open_tag=On ）。
  • CTF题目场景：某LFI题目后端代码为 include($_GET[‘file’]); ，且 allow_url_include=On ，构造URL： ?file=php://input ，同时POST提交数据 ，执行命令获取flag。

2.5 data:// 协议实战案例

  • 使用条件：开启 allow_url_include=On ，后端用 include / require ，且未过滤 data: 关键字。
  • 常见绕过技巧：
  • 编码绕过关键字过滤：若过滤了 data: ，可尝试 data:text/plain;base64, （完整格式），或用大写 DATA:// （部分服务器不区分大小写）。
  • CTF题目场景：某题目后端为 include($_POST[‘file’]); ，且 allow_url_include=On ，POST提交 file=data://text/plain;base64,PD9waHAgc3lzdGVtKCdjYXQgL2ZsYWcnKTs/Pg== ，解码后代码执行命令，获取flag。

2.6 phar:// 协议实战案例

  • 使用条件：PHP版本≥5.3.0（默认支持），后端允许传入含 .zip / .tar 等后缀的路径，可利用“phar文件后缀不影响解析”的特性。

  • 常见绕过技巧：

  • 后缀绕过：若题目限制文件后缀为 .png ，可将phar文件改名为 test.png ，构造路径 phar://test.png/test.php （仍能解析phar内的PHP文件）。

  • CTF题目场景：某题目仅允许上传 .jpg 文件，且后端用 include($_GET[‘file’]); ，先制作含恶意PHP代码的phar文件，改名为 shell.jpg 上传，再构造URL： ?file=phar://shell.jpg/shell.php ，执行代码。
