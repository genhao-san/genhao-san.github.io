真实复现DMZ区攻防场景与排查思路

发表于2026-01-29|更新于2026-01-29|应急响应
|总字数:3.4k|阅读时长:11分钟|浏览量:

前言

本次主要通过州学弟的https://mp.weixin.qq.com/s/3PjV7WIAM1C7Yu7cSLqUcA的文章来复现答题并学习DMZ区攻防场景与排查思路,同时借鉴了buutt3rf1y佬的文章来学习https://buutt3rf1y.github.io/2025/12/31/%E7%AC%AC%E4%BA%8C%E5%B1%8ASolar%E6%9D%AF%E5%BA%94%E6%80%A5%E5%93%8D%E5%BA%94%E6%8C%91%E6%88%98%E8%B5%9BWP/#%E4%BB%BF%E7%9C%9FDMZ%E7%8E%AF%E5%A2%83%E5%BA%94%E6%80%A5%E5%93%8D%E5%BA%94

题目描述

本次环境取自于实际应用中dmz隔离，仿真模拟了(生产车间/测试车间)环境，其中Windows server 2019系统为主要突破点，此机器搭建了一个对外开放的论坛，以便于让员工、客户等能够及时看到企业动态

由于企业对于网络安全疏忽，未对生产车间及其它区域服务器做物理隔离，导致攻击者以Windows server 2019系统(DMZ1)为突破口攻击成功，获取终端权限后，攻击者未收手，而是进行内网漏洞扫描，拿下另一台Ubuntu(DMZ2)的机器，并做了权限维持操作，请根据题目描述依次排查进行学习

注意：这些攻击都是在无安全设备的情况下进行的，所以在实战中遇到需根据日志及可能存在的漏洞去判断、测试、复盘等条件总结

注意：web日志与系统日志有时区差别，这也仿真了在实战中一些开放配置不当导致的溯源难度加大问题

题目列表

题目1. 漏洞排查

描述：根据开放服务排查审计日志，提交攻击者利用漏洞传入webshell的url，提交示例：flag{/flag/abc/kk=abc}

这种首先可以想到web服务，所以可以先排查端口，像一些常见的web服务端口。

  • ‌80端口‌：这是HTTP（超文本传输协议）的默认端口，用于明文传输网页内容，是访问绝大多数网站的基础端口。‌12
  • ‌443端口‌：这是HTTPS（HTTP over SSL/TLS）的默认端口，为HTTP提供加密保护，能有效防止数据被窃听或篡改，是现代网站安全传输的标准配置。‌12
  • ‌8080端口‌：作为HTTP的常用备用端口，广泛应用于Web应用开发和测试，例如Java的Tomcat、Spring Boot框架，以及一些代理服务器（如Squid）。‌35
  • ‌8000-9000端口范围‌：这些端口常被开发者用于自定义Web服务的测试和部署，例如Python的Django或Flask框架在开发时默认可能使用8000端口。‌67
  • ‌3000端口‌：在前端开发和Node.js应用中非常常见，是Vue CLI、Create React App、Vite等前端脚手架以及Express、Koa等Node.js框架的默认开发服务器端口。‌7
  • ‌5000端口‌：常用于Python Web框架（如Flask、Django）的本地开发和调试，也作为部分后端服务的默认端口。‌

题目2. Windows defender专项

描述：提交Windows defender病毒和威胁防护中，拦截攻击者最早执行的命令，提交示例：flag{dir}

本次复现学习时间可能过了一段时间了所以在defender里面查看威胁记录时好像没有了

这里面都没有找到执行的命令，那就只有使用工具

贴一个州佬那的小知识

事件 ID 含义 说明
1116 检测到恶意软件 扫描或实时防护发现了病毒/恶意软件/PUA（潜在不需要的应用程序）。
1117 执行了保护操作 系统成功采取了措施（如隔离、删除、清理）。
1118 保护操作失败 系统尝试处理威胁但失败（通常需要人工干预）。
1006 恶意软件检测 与 1116 类似，通常指防病毒引擎的检测事件。
1007 采取了行动 与 1117 类似，指防病毒平台已执行清理/隔离动作。
1015 检测到可疑行为 启发式分析检测到可疑活动（通常在尚未确认具体病毒签名时）。

题目6. 远控专项

描述：审计系统日志，提交攻击者远控后关闭Windows defender的时间，可使用桌面工具FullEventLogView辅助审计，提交示例：flag{2025/1/1 12:01:01}

事件 ID 日志类型 (Log Name) 含义 (Meaning) 关键取证细节 (Key Forensics)
5001 Windows Defender/Operational 实时保护已禁用 (Real-time Protection Disabled) 这是最核心的“结果”日志。 当看到此 ID，说明防护已关闭。 日志内容通常为：“Microsoft Defender Antivirus Real-time Protection has been disabled.”

题目7. 远控专项

描述：审计系统日志，提交攻击者创建的用户名及远程登录IP及时间，提交示例：flag{user&1.1.1.1&2025/1/1 12:01:01}

先查看一下全部用户net user

$system用户很可疑，正常系统的$符号是在右边

因此可以确定用户是$system，继续排查事件，事件ID4624是用户成功登录到计算机/域。因此用日志工具筛一遍

排序后第一条就是最早登录成功的

内网渗透扫描工具一般就有fscan、frp,刚好工具有everything我们去搜一下即可

题目12. 内网渗透排查

描述：开始排查Ubuntu(DMZ2)环境，通过前面排查的内网扫描结果以及攻击者上传的工具，攻击者对于内网机器Ubuntu(DMZ2)进行了漏洞利用，根据相关线索本地访问相关端口，攻击者为了权限维持，后期进行获取更多信息，提交攻击者在web端新增的账号，提交示例：flag{user}

扫出来有漏洞，我们可以去搜索这个漏洞了解一下

默认密码账号也可以看到nacos/nacos

查看到内网的ip直接凭借路径ip:port/nacos访问即可，再用默认账号密码登陆即可。

可以看到system

题目15. 安全加固

描述：清除攻击者在服务器新增的用户名所有信息，前往/var/flag/2文件中读取flag并提交

先userdel sys-update把用户删了，再查看/etc/passwd可以看见有伪装家目录/var/tmp/.sys，直接rm -rf /var/tmp/.sys，即可删除，再用vim编辑器，删除/etc/passwd里面对应的行即可，查看flag即可

题目16. 安全加固

描述：当前web端存在漏洞，先停止此web服务进程后，前往/var/flag/3文件中读取flag并提交

ps -ef | grep nacos查看进程ID

kill 1017终止进程,随后查看cat /var/flag/3

题目17. 安全加固

描述：攻击者通过web漏洞拿到了root账号密码，请修改密码后，前往/var/flag/4文件中读取flag并提交

passwd root改密码即可
